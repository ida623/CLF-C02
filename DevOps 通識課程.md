# DevOps 通識課程核心概念複習題

## 1. DevOps 起源與烏卡時代背景

本章介紹 DevOps 出現的歷史背景、痛點，以及它如何協助企業應對快速變化的商業環境。

1. `___`：源自軍事界，由易變性（Volatility）、不確定性（Uncertainty）、複雜性（Complexity）與模糊性（Ambiguity）四個字組成，用來描述當今快速演進、不斷變化的商業世界。
2. `___`：被稱為 DevOps 之父，他在 2007 年發現開發與運維因各自只關心自身利益（開發用敏捷追求快速交付、運維用傳統方法天天救火維持穩定）而產生嚴重衝突，進而提倡換位思考與新型協作方式。
3. `___`：在 2009 年的 Velocity 研討會上，由 Flickr 工程師發表的著名演講主題，分享他們如何打破開發（開發新功能）與運維（確保系統穩定）的衝突，透過工具與文化降低變更風險。
4. DevOps 的核心起源並非來自學術理論，而是從 `___` 中學習得來的，且大量融合了敏捷（Agile）與精實（Lean）的思想。

<details>
<summary>答案</summary>

1. VUCA（烏卡）
2. Patrick Debois
3. 一天部署超過 10 次（10+ Deploys per Day）
4. 實戰經驗

</details>

## 2. 大家認為 DevOps 是什麼

本章釐清各家廠商（Wiki、微軟、Atlassian、AWS）對 DevOps 的定義，並打破常見的工具迷思。

1. 根據 Wiki 與各家定義，DevOps 與 `___` 是互補的，兩者互相搭配才能發揮高效；而在 soft skill 方面，特別強調同理心、授權（Empowerment）與跨團隊協作。
2. 微軟認為 DevOps 是要以 `___` 為中心，不斷為其提供價值、妥善回應需求並快速達成業務目標，而非單純用來炫技。
3. AWS 在定義中特別強調上線後的 `___`，將收集到的資料反饋給開發進行調整，補齊了完整的反饋迴圈（Feedback Loop）。
4. 核心觀念總結：DevOps 主要是和工程實踐、`___` 和 `___` 有關，它絕對不等於工具。
5. 許多組織常將 DevOps 誤認為只是「工具」，是因為引進工具不需要進行高風險的 `___`、不需要老闆強烈支持（部門有錢即可買），且比虛無縹緲的心態更容易去學習與衡量。

<details>
<summary>答案</summary>

1. 敏捷開發（Agile）
2. 客戶
3. 監控（Monitoring）
4. 文化、價值
5. 組織重整（或變革）

</details>

## 3. DevOps 的目標與度量指標

本章說明如何衡量 DevOps 的實施成候，並介紹著名的度量指標。

1. DevOps 的終極目標是：快速交付 `___` 的產品。它在意的兩大面向即是「時間」與「品質」。
2. `___`：管理學大師彼得·杜拉克的名言，強調在 DevOps 落地過程中找到指標進行度量的重要性。
3. `___`：從想法的提出（點子產生），到真的交付交給客戶的這段總時間。
4. `___`：從想法開始被實踐（動工開發），到交付給客戶的這段時間。
5. `___`：計算花在修復 bug 或處理錯誤需求等「浪費」事情上的時間，占整體工作時間的比例。比例越高代表產品品質越差。

<details>
<summary>答案</summary>

1. 高品質可運行
2. 如果你無法度量，你就無法管理
3. Lead Time（交付前置時間）
4. Cycle Time（週期時間）
5. 返工指標

</details>

## 4. 落實 DevOps 的基礎：三步工作法

本章深入探討《鳳凰專案》一書中提到的三大核心原則。

### 第一步：流動原則
1. 品質管理大師戴明指出，大多數的失敗來自於 `___` 的問題，管理層的責任是改善它，而非責備個人。
2. `___`：一種用來呈現目前 high-level 工作流程的工具，會度量每個步驟的時間與步驟間的「等待時間」，用以從全局觀點找出流程瓶頸，避免局部優化。
3. 工程師的持續改善地圖則是利用 `___`（從 Code Check-in、Review 到測試發佈），藉此了解工程實踐的落實狀況。
4. 加快流動的四個方向：讓工作可視化、`___`、限制同時進行的工作量（WIP），以及持續識別浪費。

### 第二步：反饋原則
5. `___`：一個幫助決策的概念框架，將環境分為複雜、繁雜、簡單、混沌。敏捷與 DevOps 主要在解決「複雜」環境的問題。
6. 傳統 Waterfall 的回饋往往在專案末期，而 DevOps 時代則是做到 `___` 就能獲得反饋，將反饋時間縮到最短。
7. 比照豐田生產線的 `___` 機制，在軟體開發中，一旦部署流水線（Build）發現問題就必須立刻停下來修復，確保隨時都有高品質產品。
8. 顧好源頭能幫助下游，其做法包含實例化需求、`___`（Pair Programming）、持續整合與自動化測試。

### 第三步：持續學習與實驗原則
9. 根據組織文化分類，存在大量恐懼、會隱瞞真相的叫病態型；規則僵化、穀倉效應嚴重的叫官僚型；積極探索、共同承擔責任的叫 `___`。
10. Etsy 的工程師提到，要建立安全的文化，必須 `___`，消除恐懼才能讓大家講真話、及早知道壞消息。
11. 團隊可以透過每日立會、`___`、持續整合以及安排固定比例（如 10%~20%）的時間來償還 `___`，將日常工作的改進做到制度化。
12. `___`：指透過不斷的擾動、遭受破壞卻不會損壞，反而使系統變得更強大優異的特性。Netflix 便是透過 `___` 在 Production 環境進行精心的破壞性實驗來實踐此概念。

<summary>答案</summary>

1. 系統或是流程
2. 價值流程圖（Value Stream Mapping）
3. 部署流水線（Deployment Pipeline）
4. 減少批量的規模（小批量）
5. Cynefin framework（肯尼芬框架）
6. 完成一個功能之後
7. 安東繩（Andon Cord）
8. 搭檔編程（或 雙人編程）

10. 停止責備大家（Blameless）

## 5. 轉型注意事項與常見雷區

本章引述《跨越鴻溝》與 Jez Humble 的觀點，分享導入 DevOps 的策略與業界反例。

1. 根據技術採用生命週期，在團隊中要推廣新實踐時，應先找出 `___` 與 `___` 作為支持你的樁腳與側翼，並利用「從眾效應」贏得沉默大多數的支持。
2. 轉型初期應成立專門的 `___`，成員需包含不同領域知識且人脈好的人，而顧問僅扮演輔導角色。
3. Jez Humble 提出的轉型雷區包括：炒掉系統管理者或測試人員、急於進行大規模組織重整、重寫系統盲目上雲，以及 `___`（誤以為買齊工具就解決問題）。
4. 許多人常拿「法規嚴格、不是 SaaS/雲端產品、有 Legacy 系統、員工能力不足」當藉口，但 Jez Humble 認為真正阻礙轉型的原因是 `___` 與 `___`。
5. `___` 的案例證明：透過 TPS 改變系統與流程，同樣一群被認為能力不足的員工，在短短三個月內就能做出近乎完美的汽車。

## 6. 組織結構、人才類型與技術實踐

### 組織與角色
1. `___`：指出系統的設計會反映出組織本身的溝通結構（系統架構跟團隊結構是一致的）。
2. 相比於傳統的職能型與矩陣型組織，Agile 與 DevOps 更推薦 `___`（特性團隊），其由不同職能角色組成，能獨立完成某個功能，並鼓勵團隊自主學習不熟的領域。
3. 組織需要的人才模型演進：從單一領域的專家型，到一專多能的 T 型人，再到跨多個領域且皆有深入了解的 `___`。
4. 為了打破開發與運維的圍籬，書籍建議的三種做法：創建 `___`（Ops 提平台工具，RD 自己做）、Ops 融入產品團隊，以及替團隊指派運維聯絡人。

### 技術實踐：假設驅動開發、持續整合、持續發佈
5. `___`：將新產品的解法視為一種「假設」，在開發前先思考想達到的效果與需收集的數據，並透過 A/B Testing 等手段快速驗證。其核心思維是：我們要的是 `___`（成果/價值），而不是 Output（產出）。
6. `___` 提出的持續整合三大檢查問題：RD 是否每天至少提交一次代碼？每次提交是否自動觸發 Build 與自動化測試？失敗時是否能在 `___` 分鐘內修復？
7. `___` 階段主要處理開發人員之間的代碼整合；`___` 則是開發驗收完後交付給客戶（手動或自動）；`___` 則是自動將做完的系統部署到客戶的生產環境。
8. 想要做到頻繁交付，可以利用精實創業（Lean Startup）的 `___` 迴圈來打造對的東西、採用小批量開發降低風險，並利用 `___` 與燃燒圖即時顯示專案真實狀態。

### 技術實踐：測試自動化與部署策略
9. 敏捷開發中，測試的演進從前期的除錯、證明、破壞，演進到現代以 `___` 為主，企圖在早期就透過活動維持品質。
10. `___`：在開發早期利用「範例」來說明需求，讓 PM、RD、QA 達成共識，並將範例直接作為測試個案（Test Case）。
11. `___`（TDD）：不是測試方法，而是一種設計系統的方法。其步驟為先寫測試程式，再撰寫產品程式使其通過，並在保護網下進行重構。
12. `___`：一種不預先開立測試個案的手動測試，採迭代方式進行，邊使用系統邊學習、邊開測。
13. `___` 理論：指出自動化測試的投資策略，應以穩定、成本低的單元測試（Unit Test）為最多，而最不穩定的 GUI 自動化測試應該最少。
14. `___`：屬於環境面的部署策略，機器分為兩批，平時僅一批運作，新版本裝在另一批，測試完後直接切換流量，達到零停機（Zero Downtime）發佈。
15. `___`：先部署一小塊區域（或 2% 用戶）探路，運作一段時間沒問題後，再擴大部署到其餘區域。
16. `___`：在不更改或少改代碼的狀況下，利用控制端動態開關某些功能，可以讓特定客戶看到新功能，或在有危險時立即關閉功能。
